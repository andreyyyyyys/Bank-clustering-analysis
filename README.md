> Поведенческий анализ банковских транзакций для выявления групп клиентов с высоким риском оттока и разработки стратегий удержания

---

## О проекте

Проект представляет собой **кластерный анализ** клиентов банка на основе их транзакционного поведения. Цель — сегментировать клиентов по степени лояльности, выявить группы риска и предложить адресные меры удержания.

### Основные этапы:
1. **Предобработка данных** — преобразование типов, очистка, проверка пропусков
2. Исследовательский анализ данных, визуализация распределений
3. Создание агрегированных признаков на уровне клиента (RFM-подход)
4. **Стандартизация** — масштабирование признаков для корректной работы алгоритмов
5. **Кластеризация** — KMeans с оптимальным числом кластеров (метод локтя, дендрограмма)
6. **Интерпретация** — профилирование кластеров, анализ оттока и удержания

## Данные

**Источник:** `bank_transactions_data_2.csv`  
**Количество записей:** 2513 транзакций  
**Количество клиентов:** 500 клиентов  
**Период:** с 2023-01-02 по 2024-01-01

### Исходные признаки:

| Признак | Описание |
|---------|----------|
| `TransactionID` | Уникальный буквенно-цифровой идентификатор транзакции |
| `AccountID` | Уникальный идентификатор счета (несколько транзакций на счет) |
| `TransactionAmount` | Сумма транзакции (от мелких повседневных расходов до крупных покупок) |
| `TransactionDate` | Временная метка транзакции (дата и время) |
| `TransactionType` | Тип транзакции: 'Credit' (пополнение) или 'Debit' (списание) |
| `Location` | Географическое местоположение транзакции (город в США) |
| `DeviceID` | Буквенно-цифровой идентификатор устройства, с которого совершена транзакция |
| `IP Address` | IPv4-адрес, связанный с транзакцией (может меняться для некоторых счетов) |
| `MerchantID` | Уникальный идентификатор торговой точки (показывает предпочтения клиента) |
| `AccountBalance` | Баланс счета после транзакции (логически связан с типом и суммой) |
| `PreviousTransactionDate` | Временная метка предыдущей транзакции (для расчета частоты) |
| `Channel` | Канал проведения транзакции (Online, ATM, Branch) |
| `CustomerAge` | Возраст клиента (логически сгруппирован по профессии) |
| `CustomerOccupation` | Профессия клиента (Doctor, Engineer, Student, Retired) |
| `TransactionDuration` | Продолжительность транзакции в секундах (варьируется по типу) |
| `LoginAttempts` | Количество попыток входа перед транзакцией (высокие значения указывают на аномалии) |

### Агрегированные признаки

| Признак | Описание |
|---------|----------|
| `Recency` | Количество дней с последней транзакции |

| `Sum_Amount` |  Общая сумма всех транзакций |
| `Avg_Amount` |  Средняя сумма транзакции |
| `Std_Amount` |  Стандартное отклонение суммы (волатильность) |
| `TransactionDuration` |  Средняя продолжительность транзакции |
| `LoginAttempts` | Среднее количество попыток входа |
| `DeviceID` | Количество уникальных устройств |
| `IP Address` | Количество уникальных IP-адресов |
| `CustomerAge` | Средний возраст клиента |
| `CustomerOccupation` | Наиболее часто встречающаяся профессия |

### Метрики оттока (Churn) и удержания (Retention)

**Определение оттока:**  
Клиент считается ушедшим (churn), если с его последней транзакции прошло **более 60 дней**.  
Порог выбран на основе классических банковских практик и распределения Recency в данных.

**Формулы:**

| Метрика | Формула | Описание |
|---------|---------|----------|
| `churn_rate` | `(cluster_data['Recency'] > 60).mean()` | Доля клиентов, не совершавших транзакций более 60 дней |
| `retention_rate` | `(cluster_data['Recency'] <= 60).mean()` | Доля активных клиентов (транзакция была в последние 60 дней)

## Стратегии удержания по кластерам

### Кластер 1 — Лояльные
**Цель:** увеличить LTV

**Демографическое обоснование:** 29% студентов + 29% инженеров — активная аудитория с высоким транзакционным потенциалом

**Потенциальные решения:**
- Реферальная программа
- Статусная система с отображением прогресса в приложении
- Ранний доступ к новым продуктам (инвестиции, копилка)


### Кластер 2 — Высокий риск
**Цель:** вернуть активность и снизить отток

**Демографическое обоснование:** 49% студентов — мобильная, чувствительная к цене аудитория, склонная к переключению между банками

**Потенциальные решения:**
- Push-уведомления с триггерными бонусами
- Сезонные студенческие акции (доставка, такси, развлечения)


### Кластер 0 — Средний риск
**Цель:** стабилизировать поведение и повысить частоту транзакций

**Демографическое обоснование:** 33% инженеров — аудитория с устойчивым доходом, но не сформированной привычкой использования банка как основного

**Потенциальные решения:**
- Бонус за подключение автоплатежей (ЖКХ, связь)
- Дополнительный кэшбэк за регулярные операции


### Кластер 3 — Стабильные
**Цель:** увеличить частоту взаимодействия и укрепить лояльность

**Демографическое обоснование:** 37.6% пенсионеров — возрастная аудитория с потребностью в простом интерфейсе и понятных выгодах

**Потенциальные решения:**
- Адаптивный интерфейс: крупный шрифт, голосовой помощник
- Кэшбэк на приоритетные категории (аптеки, продукты, ЖКХ)
